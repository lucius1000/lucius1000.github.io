RAG是什么?

发表于 2025-09-09 更新于 2026-06-10 分类于 computer 阅读次数：

1.1.RAG基本概念

​ RAG通常指(Retrieval Augmented Generation)，即检索增强生成。这是一种结合了信息检索与语言生成技术的方法，在自然语言处理领域有广泛应用。

1.2.为什么需要RAG？

​ 目前市面上有很多通用大模型，能回答生活中的很多问题，但是对于一些有行业深度的内容回答不了。

比如我问大模型：“你了解我们公司的某某产品吗？”。

那怎么解决呢，把公司某个产品使用手册上传给大模型不就好了吗？

但会产生以下问题:

1.模型上下文长度不够，无法读取所有内容。

2.模型推理成本高，耗费资源。

3.模型推理慢

这么看来直接把文档发给模型是行不通的，所以我们只需要把问题在文档中对应的相关内容发给模型就解决了。

此时，RAG技术登场了！

1.3.RAG基本流程

分为俩阶段，索引阶段和检索阶段。

索引阶段：

步骤：

1.数据清洗和分块：对企业文档、PDF文档、数据库内容、HTML页面等原始数据进行处理。先去除噪声、HTML标签和多余空格，完成数据清洗；再将清洗后的文档划分为有意义的文本块，为后续处理做准备。

2.向量化处理：使用embedding（嵌入）模型，将步骤1得到的每一个文本块，转换为对应的向量表示，把文本语义信息转化为计算机可处理的数值向量。

3.存入向量数据库：将步骤2生成的文本块向量，和原始文本块内容一一对应，一起存入向量数据库，完成知识库的构建，为后续检索做准备。

检索阶段：

1.查询内容向量化：接收用户查询后，使用和知识库构建阶段相同的Embedding模型，将用户的问题文本转换为向量表示。

2.相似度计算：用余弦相似度、欧几里得距离或皮尔逊相关系数等方法，计算用户查询向量与向量数据库中所有文本块向量的相似度。

3.检索相关文本块：根据计算出的相似度，从向量数据库中返回Top-k个最相关的文本块。

4.提示词增强：编写提示词，将用户的原始查询问题和检索到的Top-k个相似文本块整合在一起，形成包含上下文信息的增强提示词。

5.调用大模型：把整理好的增强提示词发送给大语言模型。

6.生成最终答案：大语言模型结合提示词中的用户问题与检索到的上下文信息，生成并返回最终的回答。

1.4.RAG与传统搜索引擎的区别

1.4.1.传统搜索引擎步骤

传统搜索引擎分为俩步骤离线数据准备和在线检索响应。

一、离线阶段

1.网页抓取（爬行）

搜索引擎的爬虫程序，从种子URL出发，按照规则遍历互联网，发现并下载网页内容，同时遵循 robots.txt 协议、处理URL去重和抓取优先级调度。

2.网页预处理

对抓取的网页进行清洗与解析：去除 HTML 标签、提取正文文本、进行中文分词、过滤停用词（如 “的”“了”），并提取标题、关键词、链接结构等关键信息。

3.构建倒排索引

将预处理后的关键词与对应网页建立映射关系，生成倒排索引（记录每个关键词出现在哪些网页、位置及权重），并存储在大规模数据库中，实现**“词→文档”**的快速定位。

二、在线阶段

用户查询处理

接收用户的搜索请求，对查询词进行分词、纠错、意图识别等处理，将用户问题转化为可匹配的关键词组合。

索引匹配与召回

根据处理后的关键词，查询倒排索引，快速召回所有包含这些关键词的候选网页，形成初步结果集。

结果排序（Rank）

结合网页权重（如 PageRank）、关键词匹配度、页面质量、用户行为数据等多维度算法，对候选网页进行打分排序，确定结果的展示顺序。

结果生成与展示

为排序后的网页生成标题、摘要、缩略图等信息，并按规则分页展示给用户，同时支持广告混排、个性化推荐等附加功能。

1.4.2.区别

对比维度 传统搜索引擎 RAG 检索
核心技术原理 基于关键词 / 倒排索引的字面匹配（如 BM25、TF-IDF），依赖词频、位置等统计信息；对同义词、多义词、语境理解弱，存在 “语义鸿沟”。 基于向量语义匹配，用 Embedding 模型将文本转为向量，通过相似度计算找到语义相关内容；能理解深层语义、上下文关联，即使不包含相同关键词，也能识别相关内容。
目标与输出形式 核心目标是帮用户 “找到信息”，输出是**网页链接、文档列表 + 摘要 **，用户需要自己点开链接、阅读并整合信息，自己判断答案。 核心目标是帮用户 “得到答案”，先检索再由大模型整合信息，直接输出**连贯、定制化的自然语言回答 **，用户无需再自己处理信息。
处理流程与交互方式 是 “单次检索 + 排序” 的单向流程，不支持对话式交互，无法根据上下文理解用户的多轮提问。 是 “检索→增强提示词→大模型生成” 的闭环流程，天然支持对话式交互，能结合上下文理解用户意图，生成更贴合的回答。
适用场景 适合公开、海量互联网信息的快速导航，比如查新闻、找资料、搜公开数据。 更适合企业内部知识库、私有文档、专业领域知识的问答，比如内部手册、产品文档、病历等，能保证答案的准确性和时效性，也更容易控制信息来源。
