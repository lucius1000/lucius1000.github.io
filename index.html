如何在同一个ip和端口上部署多个网站

发表于 2026-07-04 更新于 2026-07-05 分类于 computer

1.以http的方式部署

1.1.请求头

请求头（Request Header）位于 HTTP 请求第一行之后、空行之前，是元数据，描述本次请求、客户端、资源、权限等信息。所有请求方法（GET/POST/PUT 等）都能携带，数据量很小。

客户端标识类

Header 字段 作用
User-Agent 告诉服务器客户端类型：浏览器、手机、爬虫、小程序，服务器可做适配（返回移动端页面 / 拦截爬虫）
Accept 客户端能接收的响应数据格式，例：application/json,text/html，服务器优先返回匹配格式
Accept-Encoding 支持的压缩格式（gzip/deflate/br），服务端压缩返回，减少流量
Accept-Language 客户端偏好语言 zh-CN,zh;q=0.9，网站根据这个返回中文 / 英文页面

链接与跨域类

Header 字段 作用
Host 指定目标服务器域名 + 端口，一台服务器绑定多域名时靠它区分站点
Origin 跨域请求时携带，标识请求来源页面域名，服务端 CORS 根据它判断是否允许跨域
Referer 上一页页面地址，用于统计来源、防盗链；可被前端篡改
Connection keep-alive：复用 TCP 连接，多次请求不用重复握手；close 请求完断开

缓存控制（前端性能）

Header 字段 作用
Cache-Control 控制缓存策略：no-cache 强制校验缓存、max-age=3600 缓存 1 小时
If-Modified-Since 带上上次资源修改时间，服务端对比：未修改返回 304，不返回资源本体，节省带宽
If-None-Match 携带资源 ETag 标识，资源无变化返回 304

身份鉴权

Header 字段 作用
Authorization 身份凭证，两种常用：Basic 账号密码、Bearer Token（JWT 登录令牌）
Cookie 服务器之前下发的 Cookie，自动携带；存储会话 ID、登录状态、用户偏好
X-CSRF-Token 防跨站请求伪造，POST / 修改接口必填校验

请求实体描述

Header 字段 作用
Content-Type 指定请求体的数据格式，服务端靠它解析 body：1. application/x-www-form-urlencoded：普通表单2. application/json：JSON 接口主流3. multipart/form-data：上传文件4. text/plain：纯文本
Content-Length 请求体字节长度，服务端判断何时接收完毕
Content-Encoding 请求体本身做了压缩（极少用，一般是响应压缩）

自定义请求头

Header 字段 作用
X-Request-Id 全链路追踪 ID，排查日志
X-Token 部分自定义登录令牌（标准推荐用 Authorization）
X-Platform 标识客户端平台（ios/android/web）

1.2.请求体

请求体（Request Body）空行之后的实体数据，仅POST/PUT/PATCH 等带数据的方法才有；GET 请求规范上不能带请求体，浏览器、nginx、很多服务会忽略 GET body。用来传输业务真实数据（表单、JSON、文件二进制）。

application/x-www-form-urlencoded 表单格式

  • 字段：业务表单键值对，用于简单登录、提交表单
  • 限制：不支持复杂嵌套、无法传文件

application/json（前后端分离最常用）

  • 字段：JSON 对象，支持嵌套数组、对象，适合复杂接口
  • 所有字段都是业务参数，后端直接解析成实体类

multipart/form-data 文件上传

分隔符分割普通文本字段 + 文件二进制流

1.3.部署

**原理：**HTTP1.1 协议规定浏览器请求时会携带 HOST 请求头，Web 服务器（Nginx/Apache/IIS）通过读取这个 Host 字段，在同一个 IP、同一个端口（80/443）区分不同网站，行业称为基于域名的虚拟主机（名称型虚拟主机）。

流程：

目前需要部署两个网站，a.com b.com

1.假设服务器ip为 111.222.333.444

2.设置DNS，将两个域名指向服务器ip

3.使用nginx部署

Nginx 两个虚拟主机全部监听 80端口，同一个 IP，靠 HTTP 请求里的 Host 请求头分流。

两个server块都监听 80 端口，没有端口冲突，区分逻辑完全依赖 server_name 匹配客户端发送的 HOST 请求头。

注意：老旧 HTTP1.0 客户端不携带 Host，Nginx 会匹配第一个加载的 server块，导致访问错乱，所以推荐配置default_server兜底拦截。

2.以https的方式部署

2.1.介绍

HTTPS就是HTTP+TLS加密

https的默认端口是443，底层 TCP 先建立 TLS 加密通道，再传输原本的 HTTP 请求头、请求体。HTTP 原有所有请求头、请求体规则完全不变，字段作用一模一样；HTTPS 只是把整个报文加密，内容结构没改。

SNI（服务器名称指示）：客户端在 SSL 握手阶段就携带要访问的域名，Nginx 提前知道域名，选择对应证书；握手完成后再用 HTTP Host 头二次匹配站点目录

现代浏览器、Nginx 默认开启 SNI，所以能单 IP:443 跑多个 HTTPS 网站。

客户端携带SNI过程

1.DNS 解析 b.com → 服务器公网 IP；

2.客户端发起 TCP 三次握手，建立和 IP:443 的 TCP 管道；

3.TCP 通了之后，立刻发送 TLS Client Hello 握手包。

Client Hello 包含：TLS 版本、加密套件列表、随机数、扩展段 extensions，

SNI就是其中一个扩展。

注意:

1.主流浏览器访问网站时，默认强制开启 SNI，自动携带域名，无需任何配置。

2.curl命令默认携带SNI。

3.python、golang、java发起请求默认携带SNI。

2.2.部署

方案 A：两套独立证书（推荐，域名分离）

a 证书：a.crt/a.key 适配 a.com

b 证书：b.crt/b.key 适配 b.com

存放路径 /etc/nginx/ssl/

方案 B：一张多域名通配证书（省事）

申请 SAN 证书，同时包含 a.com、b.com ，所有 server 共用一套证书

A Nginx 完整配置（独立证书 + SNI）

B Nginx只用一张多域名 SAN 证书

握手时无论 SNI 是 a 还是 b，证书都能匹配，后续依旧靠 Host 区分网站目录。

2.3.访问流程

1.DNS + TCP 连接

1.浏览器解析 b.com → 服务器公网 IP

2.客户端发起 TCP 连接，目标 IP:443 ，建立 TCP 通道

2.SSL 握手（SNI 发挥作用，决定返回哪张证书）

1.客户端发送 Client Hello，携带SNI扩展字段：b.com

2.Nginx 读取 SNI 里的域名 b.com，遍历所有 listen 443 ssl 的 server 块，匹配字段 server_name b.com

3.Nginx 读取该块配置，加载 b.crt/b.key 证书，下发给浏览器

4.双方完成密钥协商、加密通道建立

关键：这一步还没有 HTTP 报文，完全靠 SNI 区分域名，选择证书

3.加密通道内发送 HTTP 请求（Host 头区分站点目录）

加密通道建立完成后，浏览器发送加密的 HTTP 请求报文：

1.Nginx 解密报文，提取 Host: b.com

2.再次匹配 server_name b.com对应的 server 块

3.使用配置里 root /www/b 目录，读取页面内容加密返回浏览器
